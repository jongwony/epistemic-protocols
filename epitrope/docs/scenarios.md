# Epitrope 사용 시나리오

> 실사용 전제 — 이 시나리오들은 사용자가 해당 프로토콜을 의식적으로 호출(`/calibrate`)하는 상황을 기술합니다.
> Skill이 세션에 로드되지 않으면 충돌이 발생하지 않습니다.

**프로토콜**: Epitrope (`/calibrate`)
**결핍 → 해소**: `DelegationAmbiguous → CalibratedDelegation`

---

### 1. "이 프로젝트 전체적으로 리팩토링 좀 해 줘" `SW`

**사용자 맥락**: 범위가 넓은 코드베이스 전반의 리팩토링 요청이지만, AI가 어디까지 자율적으로 파일을 수정해도 되는지, 아키텍처 변경까지 포함하는지, 테스트 수정은 자율적으로 해도 되는지 위임 범위가 불명확하다.

**트리거 발화**: "이 프로젝트 전체적으로 리팩토링 좀 해 줘" ("handle this" 범위 없는 위임)

**호출 예시**:
```
/calibrate 코드베이스 전반 리팩토링을 시작하기 전에, 어디까지 자율적으로 수정하고 어디서부터 확인받아야 할지 정해 줘.
```

**결핍(전)**: 다중 도메인(FileModification, Strategy, Exploration)에 걸쳐 있어 DelegationContract가 필요하다. 자율 범위 미정으로 과도자율 또는 과소자율 마찰이 예상된다.

**해결(후)**: 시나리오 기반 인터뷰로 WHO/WHAT/HOW MUCH를 계약화. 예: FileModification=Autonomous(테스트 포함), Strategy=AskBefore(아키텍처 변경), External=AlwaysAsk(PR).

**인접 프로토콜 배제**: **Telos** — 목표(리팩토링)는 이미 있고, 부족한 것은 목표가 아니라 위임 강도 조정이다.

---

### 2. 이전 과도한 변경 후 새 작업 시작 `SW`

**사용자 맥락**: 지난 작업에서 AI가 요청하지 않은 파일까지 수정하고 추가적인 "개선"을 해서 friction이 발생했다. 이번 새 작업("인증 모듈에 2FA 추가")을 시작하며, 이전 경험을 바탕으로 위임 경계를 명확히 설정하고 싶다.

**트리거 발화**: "이번에는 요청한 것만 해 줘" (prior friction 트리거)

**호출 예시**:
```
/calibrate 지난번에 요청 외 파일까지 수정해서 문제가 있었어. 이번 2FA 작업은 위임 범위를 먼저 정하고 시작하자.
```

**결핍(전)**: 이전 세션의 부정적 경험이 직접적 촉발 원인. 위임 범위가 "이전처럼 넓게"와 "요청한 것만"의 사이에서 명확하지 않다.

**해결(후)**: Solo 엔트리에서 도메인별 시나리오 질문으로 `Autonomous/ReportThenAct/AskBefore`를 고정해 재질문 없이 일관된 수행 계약을 만든다.

**인접 프로토콜 배제**: **Aitesis** — 정보 부족이 아니라 "무엇까지 자율 수행 가능한가"라는 권한 경계 불명확성이 핵심이다.

### 3. 경영 컨설팅 실사 multi-domain 위임 `KW`

**사용자 맥락**: 파트너가 고객사 실사를 AI에 위임한다. 실사가 재무/운영/법률/시장 등 다중 도메인에 걸쳐 있고, 각 영역별로 자율 수행/보고 후 진행/사전 승인이 달라야 한다.

**트리거 발화**: "이 고객사 실사를 handle 해 줘" (도메인별 자율성 수준 차별화 필요)

**호출 예시**:
```
/calibrate 고객사 실사를 맡기려는데, 재무/운영/법률/시장 각 영역별로 자율성 수준을 다르게 설정해 줘.
```

**결핍(전)**: "handle"이라는 표현에 구체적 위임 범위가 없어 DelegationAmbiguous가 발생한다. 재무/운영/법률/시장 각 도메인의 자율성 수준이 다를 수밖에 없다.

**해결(후)**: 도메인별 시나리오 인터뷰로 재무=ReportThenAct, 법률=AskBefore, 시장=Autonomous 등 세분화된 DelegationContract를 만든다.

**인접 프로토콜 배제**: **Aitesis** — "실행에 필요한 맥락 부족"이 아니라 "누가, 무엇을, 얼마만큼의 자율성으로 할 것인가"가 미정이다.

### 4. "이번 달 콘텐츠 다 맡아줘" `CC`

**사용자 맥락**: 1인 유튜버가 번아웃으로 인해 AI에게 이번 달 콘텐츠 전체를 맡아 달라고 요청했다. 기획, 스크립트, 썸네일 기획, SEO 최적화, 업로드 스케줄링 등 다영역에 걸친 위임이며, "handle this" 타입의 범위 없는 요청이다.

**트리거 발화**: "이번 달 콘텐츠 다 맡아줘" (번아웃으로 인한 포괄적 위임, WHO/WHAT/HOW MUCH 모두 모호)

**호출 예시**:
```
/calibrate 이번 달 콘텐츠 제작을 전부 맡기고 싶은데, 기획/스크립트/SEO 각각 어디까지 자율적으로 진행할지 정해 줘.
```

**결핍(전)**: WHO(AI 단독? 편집자 협업?), WHAT(어디까지?), HOW MUCH(자율 실행 vs 승인 후 실행?) 세 차원 모두 모호하다.

**해결(후)**: 시나리오 기반 인터뷰로 기획=AskBefore, 스크립트=ReportThenAct, SEO=Autonomous 등 도메인별 DelegationContract를 구성한다.

**인접 프로토콜 배제**: **Telos** — 목표(콘텐츠 제작)는 있고, 부족한 것은 위임의 범위와 자율성이다.

---

### 5. 지속 파일 vs 임시 파일 자율성 분리 `CC`

**사용자 맥락**: 리팩터 작업에서 빌드 산출물 삭제는 자유롭게 허용하지만, `CLAUDE.md`, 운영 config, 메모리 파일 수정은 사전 승인받고 싶어 한다.

**트리거 발화**: "코드는 알아서, 규칙/설정 파일은 꼭 물어봐." (FileModification persistence 분기 필요)

**호출 예시**:
```
/calibrate 리팩터 작업에서 코드 파일은 자율 수정, CLAUDE.md/config/메모리 파일은 사전 승인으로 나눠서 위임 설정해 줘.
```

**결핍(전)**: 파일 수정 정책이 단일 레벨이면 안전 자산까지 자동 변경되거나, 반대로 사소한 변경도 과도 승인 요청이 발생한다.

**해결(후)**: Epitrope가 FileModification을 ephemeral/durable로 세분해 서로 다른 autonomy를 계약에 명시하고, 리뷰 단계에서 기본값까지 투명하게 보여준다.

**인접 프로토콜 배제**: **Syneidesis** — 의사결정 갭이 아니라 권한 세분화 정책을 정하는 캘리브레이션 문제다.
