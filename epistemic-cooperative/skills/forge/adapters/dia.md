# Adapter: dia

Progressively disclosed — Read only after the `dia` adapter is selected in Phase 0. Satisfies the Vendor Adapter Contract; the core never learns these specifics.

This is forge's **constrained-substrate** adapter. Where grounding a vendor's prompt guide normally forms a prompt for a *capable* tool, `dia` projects a resolved intent into a **Dia browser custom skill** — a substrate that *cannot read your repo and lacks most agent primitives* (no shell, no file access, no subagents, no structured state). The endpoint is therefore not a one-shot "initial prompt" but a **standing custom-skill recipe** the user installs once and invokes by slash command; it is the `PromptArtifact` form for the Dia substrate. Because Dia drops primitives a capable tool would assume, this adapter additionally surfaces a **capability-gap degradation note** (via `validate`) declaring what the substrate cannot do, rather than silently approximating it.

When the Dia skill's purpose is to apply an existing protocol or procedure to Dia's context, the resolved content *is* that procedure's structure: the user supplies it directly, points to a source document, or feeds in a portable summary produced upstream by `/distill` (an optional feeder, not a dependency). Intent-resolution is then minimal and the work is the structure-preserving projection into Dia's skill form.

- **Reference**: the Dia browser (The Browser Company's Chromium AI browser) custom-skill surface. A Dia custom skill is a **conditioning instruction applied over the current page/selection/tab context**, not a chat message. Create-skill field mapping: **name** → the skill's slash command (entered *without* a leading `/`); **description** → the routing line shown in the skill list; **body** → the prompt Dia runs over the current context. Canonical coordinate: the Dia app's Skills settings (Settings → Skills → New Skill). There is no public versioned doc URL for the field set, so the reference is the app's current create-skill form rather than a fetchable spec.
- **capabilities**: a single free-text **body** that conditions Dia's answer over the current page / selection / open tabs / web / chat context, invoked by the skill's slash command. No structured mode state, no file or tool access, no subagents, no shell, no codebase read, no cross-turn store beyond the visible conversation. The body is the entire program; everything the realization needs must live in it as plain instruction.
- **fetch_guide_snapshot**: confirm the create-skill field set (name / description / body) and the conditioning-over-context semantics from the Dia app's current Skills form. There is no dated/versioned doc, so the staleness guard cross-checks the field set against the app's live form; when it cannot be confirmed, mark the draft `stale-guide` and fall back to the name/description/body mapping recorded here.
- **derive_prompt_schema**: the `DiaRecipeInstruction` schema — `{ name, description, body }` — where the **body** must: apply the intent to the current Dia context (page/selection/tabs/web/chat); use **only Dia-visible evidence**; declare the capability-gap note inline; maintain a visible **"protocol ledger"** inline in the response (Dia has no structured state, so the running state is rendered as text the next turn can read back); handle each decision point that needs the user's judgment as **context-first → concise question + numbered options → stop and wait**; continue from the visible ledger on follow-ups; and emit a **deficit → resolution trace** (or the procedure's completion trace) at the end.
- **project**: project the resolved intent through the schema into a `DiaRecipeInstruction` body that **preserves the source structure** — each procedure step becomes a body instruction; each judgment point becomes a context-first numbered-option stop; each fixed choice set becomes option labels; the completion condition becomes the visible-ledger trace; standing constraints become standing body rules. Structure-preserving, not a template that merely name-drops the procedure. `name` ← the procedure's slash command (or a chosen command) without a leading `/`; `description` ← the routing line.
- **validate**: in addition to the standard `stale-guide` / `transport-unsafe` flags, emit a **`capability-gap` degradation note** — for each Dia primitive the requested skill needs but the substrate lacks, an entry `{ primitive, why_unavailable_in_dia, declared_degradation }`. Expected entries:
  - **codebase-read (Read/Grep/Glob)** — unavailable: Dia conditions over the visible page/tabs, not a repo. Degradation: evidence is limited to Dia-visible content; the body instructs gathering evidence from the current context only (DEGRADE).
  - **shell (Bash)** — unavailable: no command execution. Degradation: no dynamic observation; observation steps fall back to asking the user or reading visible content (DEGRADE).
  - **file-write (Write/Edit)** — unavailable: no file persistence. Degradation: outputs are emitted as inline text in the response, not written (DEGRADE).
  - **subagent (Task)** — unavailable: no fresh-context dispatch. Degradation: any fresh-context verification (e.g. a refute-posture comprehension check) runs **inline and weakened** — no fresh-context isolation — and the body declares the weakened realization (DEGRADE; escalates to **FAIL** when the requested skill's completion *requires* a fresh-context verdict that cannot be faithfully simulated inline).
  - **structured cross-turn state** — unavailable: no mode state beyond the visible conversation. Degradation: state is tracked only in the visible inline ledger, which the next turn reads back (DEGRADE).
  Resolution: **FAIL** the realization (surface for Redirect / substrate change) if a gap touches the skill's core purpose; **DEGRADE** (proceed with the declared gap) if it only reduces evidence-richness or state-tracking. Surface the full capability-gap note with the draft; never present a Dia realization whose purpose-touching gap is silently approximated.
- **ingestion (manual paste only)**: Dia's skill store is **encrypted and cloud-synced with no file, CLI, or deeplink import path**. The adapter therefore produces a **paste-ready** `DiaRecipeInstruction` (name + description + body) for the user to enter by hand in Settings → Skills → New Skill. forge does not — and cannot — automate ingestion; it emits the artifact and stops (Rule 10, Formation not execution).
- **PromptArtifact**: a paste-ready Dia custom-skill recipe — `name`, `description`, and a `body` that is a **literal opaque payload** (display it verbatim, fenced, so it survives on-screen rendering) — the `PromptArtifact` form for the Dia substrate, accompanied by the capability-gap degradation note. forge emits it and stops; the user pastes it into Dia.
