# Adapter: dia

Progressively disclosed — Read only after the `dia` adapter is selected in Phase 0. Satisfies the Vendor Adapter Contract; the core never learns these specifics.

This is the **substrate-realization** adapter. Where `higgsfield` / `gpt-image` / `codex-goals` / `claude-session` ground a vendor's prompt guide to form a prompt for a *capable* tool, `dia` projects a portable epistemic essence — typically a `ProtocolEssenceIR` (a `/distill` output over a protocol SKILL.md) — into a **Dia browser custom skill**, a substrate that *cannot read your repo and lacks most agent primitives*. It is therefore forge's fullest realization of the Ektyposis `/realize` morphism: it grounds Dia's artifact form, projects each structural element of the protocol structure-preservingly, and runs `declare_unavailable_primitives` to produce an `UnavailableInDia` degradation ledger. Unlike the creative adapters, this adapter's input is a `ProtocolEssenceIR`, not a reverse-induced `ResolvedIntentIR`: the resolved content *is* the distilled protocol essence, so intent-resolution is minimal and the work is the structure-preserving projection into Dia's skill form.

- **Reference**: the Dia browser (The Browser Company's Chromium AI browser) custom-skill surface. A Dia custom skill is a **conditioning instruction applied over the current page/selection/tab context**, not a chat message. Create-skill field mapping: **name** → the skill's slash command (entered *without* a leading `/`); **description** → the routing line shown in the skill list; **body** → the prompt Dia runs over the current context. Canonical coordinate: the Dia app's Skills settings (Settings → Skills → New Skill). There is no public versioned doc URL for the field set, so the reference is the app's current create-skill form rather than a fetchable spec.
- **capabilities**: a single free-text **body** that conditions Dia's answer over the current page / selection / open tabs / web / chat context, invoked by the skill's slash command. No structured mode state, no file or tool access, no subagents, no shell, no codebase read, no cross-turn store beyond the visible conversation. The body is the entire program; everything the realization needs must live in it as plain instruction.
- **fetch_guide_snapshot**: confirm the create-skill field set (name / description / body) and the conditioning-over-context semantics from the Dia app's current Skills form. There is no dated/versioned doc, so the staleness guard cross-checks the field set against the app's live form; when it cannot be confirmed, mark the draft `stale-guide` and fall back to the name/description/body mapping recorded here.
- **derive_prompt_schema**: the `DiaRecipeInstruction` schema — `{ name, description, body }` — where the **body** must: apply the protocol to the current Dia context (page/selection/tabs/web/chat); use **only Dia-visible evidence**; declare the `UnavailableInDia` ledger inline; maintain a visible **"protocol ledger"** inline in the response (Dia has no structured MODE STATE, so the running state is rendered as text the next turn can read back); handle each Constitution gate as **context-first → concise question + numbered options → stop and wait**; continue from the visible ledger on follow-ups; and emit the **deficit → resolution trace** at convergence.
- **project**: project the `ProtocolEssenceIR` through the schema into a `DiaRecipeInstruction` body that **preserves the protocol's structure** — each morphism step becomes a body instruction; each gate point becomes a context-first numbered-option stop; each answer constructor becomes a gate option label; the convergence condition becomes the visible-ledger deficit→resolution trace; the invariants become standing body rules. Structure-preserving, not a template that merely name-drops the protocol. `name` ← the protocol's slash command without its leading `/`; `description` ← the protocol's routing line.
- **declare_unavailable_primitives → `UnavailableInDia`**: for each primitive the protocol's `tool_grounding_ops` depend on but Dia lacks, emit an entry `{ primitive, why_unavailable_in_dia, declared_degradation }`. Expected entries:
  - **codebase-read (Read/Grep/Glob)** — unavailable: Dia conditions over the visible page/tabs, not a repo. Degradation: evidence is limited to Dia-visible content; the body instructs the protocol to gather evidence from the current context only (DEGRADE).
  - **shell (Bash)** — unavailable: no command execution. Degradation: no dynamic observation; empirical-observation steps fall back to asking the user or reading visible content (DEGRADE).
  - **file-write (Write/Edit)** — unavailable: no file persistence. Degradation: outputs are emitted as inline text in the response, not written (DEGRADE).
  - **subagent (Task)** — unavailable: no fresh-context dispatch. Degradation: any fresh-context verification (e.g. a refute-posture comprehension gate) runs **inline and weakened** — no fresh-context isolation — and the body declares the weakened realization (DEGRADE; escalates to **FAIL** when the protocol's convergence *requires* a fresh-context subagent verdict that cannot be faithfully simulated inline).
  - **transcript/session stores** — unavailable: no structured cross-turn state. Degradation: state is tracked only in the visible inline protocol ledger, which the next turn reads back (DEGRADE).
- **validate**: **FAIL** the realization if an unavailable primitive touches protocol identity or a convergence condition (e.g. a protocol whose convergence *requires* a fresh-context subagent verdict cannot be faithfully realized in Dia → FAIL, surfaced for Redirect / substrate change); **DEGRADE** (proceed with the declared gap) if the primitive only reduces evidence-richness or state-tracking. Surface the full `UnavailableInDia` ledger with the draft; never present a Dia realization whose identity-touching gap is silently approximated.
- **ingestion (manual paste only)**: Dia's skill store is **encrypted and cloud-synced with no file, CLI, or deeplink import path**. The adapter therefore produces a **paste-ready** `DiaRecipeInstruction` (name + description + body) for the user to enter by hand in Settings → Skills → New Skill. forge does not — and cannot — automate ingestion; it emits the artifact and stops (Rule 10, Formation not execution).
- **InitialPrompt**: a paste-ready Dia custom-skill recipe — `name`, `description`, and a `body` that is a **literal opaque payload** (display it verbatim, fenced, so it survives on-screen rendering) — the `DiaRecipeInstruction` realization of the protocol essence for the Dia substrate, accompanied by the `UnavailableInDia` degradation ledger. forge emits it and stops; the user pastes it into Dia.
