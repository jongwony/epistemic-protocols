# Katalepsis 사용 시나리오

> 실사용 전제 — 이 시나리오들은 사용자가 해당 프로토콜을 의식적으로 호출(`/grasp`)하는 상황을 기술합니다.
> Skill이 세션에 로드되지 않으면 충돌이 발생하지 않습니다.

**프로토콜**: Katalepsis (`/grasp`)
**결핍 → 해소**: `ResultUngrasped → VerifiedUnderstanding`

---

### 1. AI가 작성한 재귀 SQL CTE 이해 `SW`

**사용자 맥락**: AI가 계층적 조직도 데이터를 쿼리하기 위해 재귀 CTE(Common Table Expression)를 작성했다. 쿼리는 정확히 동작하지만, 개발자가 재귀 종료 조건과 UNION ALL의 동작 방식을 이해하지 못한 상태다.

**트리거 발화**: "이 쿼리가 어떻게 동작하는지 설명해 달라" (사용자 주도 + 결과 R 존재)

**호출 예시**:
```
/grasp 이 재귀 CTE 쿼리가 어떻게 동작하는지, 특히 종료 조건과 UNION ALL 부분을 이해하고 싶어.
```

**결핍(전)**: 결과물 R(SQL 쿼리)이 이미 존재하고, 사용자가 이해 부족을 인지한 상태. 재귀 종료 조건과 UNION ALL 동작을 오해하면 수정 시 무한루프 위험.

**해결(후)**: 카테고리(Code/Logic) 선택 후 소크라테스식 질문으로 동작 순서와 종료 조건 이해를 검증해 VerifiedUnderstanding에 도달한다.

**인접 프로토콜 배제**: **Aitesis** — 실행 전 컨텍스트 부족이 아니라, 이미 완료된 결과물의 이해 검증이 목표다.

### 2. CQRS+이벤트 소싱 패턴 파악 `SW`

**사용자 맥락**: AI가 CQRS+이벤트 소싱 패턴으로 주문 처리 도메인을 구현했다. 코드는 동작하지만 개발자가 Command→Event→Projection 흐름과 eventual consistency 보장 방식을 확실히 이해하지 못한다.

**트리거 발화**: "이 아키텍처 패턴을 정확히 이해했는지 확인하고 싶다" (사용자 주도)

**호출 예시**:
```
/grasp CQRS+이벤트 소싱 패턴에서 Command→Event→Projection 흐름과 eventual consistency 보장 방식을 이해했는지 확인하고 싶어.
```

**결핍(전)**: 단일 코드가 아닌 아키텍처 패턴 전체의 이해가 부족. Command→Event→Projection 흐름의 오해로 잘못된 수정 위험.

**해결(후)**: Architecture/Pattern 카테고리로 Command/Event/Projection 각 단계의 이해를 검증하고, 정확히 이해된 항목만 Task 완료 처리한다.

**인접 프로토콜 배제**: **Prothesis** — 새로운 관점 분석이 아니라, 이미 완료된 AI 작업 결과물(R)에 대한 이해 검증이다.

### 3. Git rebase 충돌 해결 결과 검증 `SW`

**사용자 맥락**: AI가 복잡한 rebase 충돌을 해결했는데, 15개 파일에 걸친 변경사항 중 어떤 것이 원래 브랜치의 변경이고 어떤 것이 충돌 해결을 위한 수정인지 개발자가 구분하지 못한다.

**트리거 발화**: "이 충돌 해결에서 무엇이 바뀌었는지 이해하고 싶다"

**호출 예시**:
```
/grasp rebase 충돌 해결에서 원래 브랜치 변경과 충돌 해결 수정을 구분해서 이해하고 싶어.
```

**결핍(전)**: AI 작업의 '차이'를 이해하는 것이 핵심. 원래 변경과 충돌 해결 수정을 구분하지 못하면 잘못된 롤백 위험.

**해결(후)**: Modification/Diff 카테고리로 각 파일의 변경 이유를 검증하고, 이해가 확인된 범위만 완료 처리한다.

**인접 프로토콜 배제**: **Syneidesis** — 의사결정 갭 점검이 아니라 변경 산출물에 대한 이해 확증 과정이다.

### 4. DCF 밸류에이션 모델 산출 결과 `KW`

**사용자 맥락**: AI가 3개년 DCF 밸류에이션 모델을 완성하여 기업가치 범위(3,200~4,100억 원)를 산출했다. 투자심사역이 결과의 논리적 근거를 이해하고 싶다.

**트리거 발화**: "이 숫자가 어떻게 나온 건지 설명해 줘" (수치 모델의 가정/계산 이해)

**호출 예시**:
```
/grasp DCF 밸류에이션 결과가 3,200~4,100억으로 나왔는데, 할인율 가정과 터미널 밸류 계산이 어떻게 이 범위를 만드는지 이해하고 싶어.
```

**결핍(전)**: 결과(R)가 이미 존재하고, 사용자는 할인율 가정, 터미널 밸류 계산, 감도 분석이 어떻게 결합되어 이 범위를 만들었는지를 이해하고 싶다.

**해결(후)**: Expectation/Calculation 갭 중심 질문을 통해 할인율, 터미널 밸류, 감도 분석의 논리적 연결을 검증해 VerifiedUnderstanding에 도달한다.

**인접 프로토콜 배제**: **Telos** — 목표 설정이 아니라 결과 해석을 사용자 머릿속에 검증 가능한 형태로 정착시키는 문제다.

### 5. AI가 설계한 온라인 강의 가격 전략 3단계 티어 `CC`

**사용자 맥락**: AI가 시장 분석을 기반으로 3단계 가격 티어(기본/프로/엔터프라이즈)와 각 티어의 기능 구성을 제안했다. 강사가 티어 구성 논리를 이해하고 싶다.

**트리거 발화**: "프로 티어에 1:1 코칭을 넣고 기본에서 뺀 이유가 뭔지... walk me through 해 줘"

**호출 예시**:
```
/grasp 가격 티어 구성에서 프로 티어에 1:1 코칭을 넣고 기본에서 뺀 이유, 이 가격 차이가 전환율에 어떤 영향을 줄 것으로 본 건지 이해하고 싶어.
```

**결핍(전)**: 가격 전략의 논리를 소화하지 못한 상태. 비즈니스 전략의 논리 근거 이해가 필요하다.

**해결(후)**: Strategy/Logic 카테고리로 티어 구성 논리와 전환율 가정을 검증하고, 이해가 확인될 때 Task를 완료한다.

**인접 프로토콜 배제**: **Prothesis** — 새로운 분석 프레임이 필요한 게 아니라, 이미 완료된 작업의 이해가 목적이다.
